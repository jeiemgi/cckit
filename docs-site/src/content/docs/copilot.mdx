The copilot is cckit's prompt machine. It doesn't run agents itself — it hands the agent a clear
plan and the agent does the work. One turn of the loop fans a wave out, gates the results, and
advances to the next wave.

## One turn of the loop

<Steps>

1. **Plan** — `cckit copilot` reads the board, finds the next wave of unblocked issues, and writes a
   brief: one subagent task per issue, each in its own worktree.

2. **Fan out** — the agent (Claude Code) spawns one subagent per issue from that brief. They run in
   parallel, each implementing its issue and opening a pull request.

3. **Gate + merge** — `cckit watch --merge` checks every open pull request. The clean ones
   (passing checks, no conflicts) are merged; the rest are reported with what to fix.

4. **Advance** — merging unblocks the next wave. Re-run `cckit copilot` and the loop repeats until
   there's nothing left to do.

</Steps>

<Aside type="note" title="Why a brief, not a launcher">
cckit emits the plan as instructions and the agent enacts it. That keeps the work inside Claude
Code's own tools — Task subagents, worktrees, background runs — instead of a separate process you
can't see.
</Aside>

## The commands

| Do this | Command |
| ------- | ------- |
| Get the fan-out brief | `cckit copilot` |
| Scope to one effort | `cckit copilot --effort <N>` |
| Gate + merge the wave | `cckit watch --merge` |
| Let the captain self-pace | `cckit watch --loop` |
| Machine-readable plan | `cckit plan --llm` |
