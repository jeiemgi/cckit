## Why cckit

<CardGrid stagger>
  <Card title="One lifecycle" icon="seti:todo">
    Start an issue, get an isolated worktree and branch, open a PR, merge, clean up — each step is one command, correct by construction.
  </Card>
  <Card title="A copilot loop" icon="rocket">
    `cckit copilot` turns the board into a wave of parallel agent work; `cckit watch` gates and merges the PRs, then advances to the next wave.
  </Card>
  <Card title="Agent-operable" icon="puzzle">
    Every verb has a machine-readable mode (`cckit <verb> --llm`, uniform data as TOON) and an `AGENTS.md` contract, so agents drive the kit without scraping human output.
  </Card>
  <Card title="Zero hard dependencies" icon="approve-check">
    Pure bash with graceful fallbacks; it auto-detects optional tools and degrades instead of failing.
  </Card>
</CardGrid>

## Learn it as a visual guide

These docs are a learning book: coloured **tags** tell you how each thing is triggered (a command you type, a skill you run, something that happens just by talking to Claude), and **interactive diagrams** show each flow left to right. Start with [how to read this guide](/how-to-read/), then the [GitHub cycle](/github-cycle/).
