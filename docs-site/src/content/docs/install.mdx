Put the `cckit` command on your PATH, confirm it works, and adopt it in a repo.

cckit is pure bash over `git`, `gh`, and `jq`. There is no build step — installing it puts the `cckit` dispatcher on your PATH, the list of folders your terminal searches when you type a command. Pick whichever route fits.

## Install globally

<Steps>

1. **Pick an install method.** Each tab is one route — pick the one that fits, and copy the box.

   <Tabs syncKey="install-method">
     <TabItem label="One-liner">
       ```sh title="macOS · Linux · WSL — no npm or brew"
       curl -fsSL https://cckit.dev/install.sh | bash
       ```
     </TabItem>
     <TabItem label="Homebrew">
       ```sh title="Formula/cckit.rb"
       brew install jeiemgi/cckit/cckit
       ```
     </TabItem>
     <TabItem label="npm">
       ```sh title="the bare name is taken, so cckit is scoped"
       npm install -g @jeiemgi/cckit
       ```
     </TabItem>
     <TabItem label="From source">
       ```sh title="symlinks bin/cckit onto your PATH"
       git clone https://github.com/jeiemgi/cckit.git
       cd cckit && ./scripts/install.sh
       ```
     </TabItem>
   </Tabs>

2. **Verify it's on your PATH.**

   ```sh
   cckit version       # prints the installed version
   cckit doctor        # preflight: git/gh/jq present, gh authenticated
   ```

</Steps>

`cckit install [bin-dir]` symlinks `bin/cckit` into the first writable PATH folder (`~/.local/bin`, then `/usr/local/bin`, then `~/bin`), or a folder you pass.

It is a **symlink, not a copy** — the dispatcher resolves the link back to the repo, so `git pull` upgrades your install with no reinstall.

## Dependencies

| Required | Why |
| --- | --- |
| `git`, `gh`, `jq` | the lifecycle (worktrees, GitHub, JSON) |
| `bash` 3.2+ | the dispatcher + lib |

Optional, auto-detected, never required:

| Optional | Enables |
| --- | --- |
| `tmux` | `cckit orchestrate` (parallel panes) |
| `glow` / `fzf` / `gum` | nicer terminal output and pickers |
| Node + Chrome | `cckit debug` (browser debugging via chrome-devtools-axi) |

`cckit doctor` reports what's missing.

<Aside type="tip" title="What is a symlink? (skip if you know)">
A symlink is a pointer file: it stands in one folder but redirects to a file somewhere else. cckit installs as a symlink into a PATH folder that points back to the cloned repo, so updating the repo updates your installed command automatically.
</Aside>

## Set up in a repo

Installing puts `cckit` on PATH globally. **Adopting it in a repo** is a separate, per-repo step:

```sh
cckit scan          # what's here already
cckit init          # scaffold cckit.config.json + .claude/ (greenfield)
```

For a repo that already has kit-shaped files, see the [adoption flow](/adoption/).

## Claude Code (IDE) config

cckit ships as a Claude Code plugin (`.claude-plugin/plugin.json` + `commands/` + `skills/`). With the repo present, Claude Code discovers the `/kit-*` slash commands and skills.

`cckit init` scaffolds a project's `.claude/` folder — agents, hooks, rules, settings — from a profile, so the editor picks up the kit's conventions.

Other agents drive the same CLI through [AGENTS.md](https://github.com/jeiemgi/cckit/blob/main/AGENTS.md) — no IDE required.

## Safety and permission gate

Every contribution and release runs the local gate first:

- **`scripts/check.sh`** — shell syntax, valid manifests, no stray branding.
- **The secret + privacy guard** (`scripts/lib/secret-guard.sh`) — blocks env files, keys, tokens, and a user-declared `.cckit/privacy-denylist`.

Nothing publishable leaves without passing the gate. The git hook in `githooks/pre-commit` wires the same guard at commit time.
