Plain-language explainer · updated for the Agentation direction

Point at a component, tell Claude what to fix

kit-annotate lets you click any component in your running dev app, leave a note, and have Claude read a precise, structured record and fix the code. It's wired into claude-kit for Claude Code, and it runs on a maintained open package called Agentation rather than something we hand-build. Here's the whole thing in your language.

1What it does

Today you tell Claude "the spacing on the admin card is too tight" and it guesses which file. With kit-annotate you instead click the actual component, type the note right there, and Claude receives: the component (e.g. <AdminCard>), its place in the tree, a source-file pointer, and your comment. Then it opens the code and fixes it.

≈

Figma comments, but pinned to React components instead of pixels — and the teammate who resolves them is Claude, working in your repo.

2How the plan evolved (so the docs make sense)

We considered three approaches before landing here. Knowing the path explains why the final design is shaped the way it is:

IdeaVerdict
Extend onUI (a DOM-annotation extension you'd installed)❌ Closed, version-pinned, DOM-only — couldn't model components.
Build our own Chrome extension❌ The "two worlds" problem, an install, a daemon, React-19 source headaches.
An in-app wrapper — and adopt Agentation✅ A maintained, open package already does this well. We wire it to Claude.
✦

An in-app wrapper (a React component you render) has full, direct access to React — no extension sandbox, no install, and source location comes from your dev build, not a browser hack. Agentation is exactly that wrapper, already shipping, so we adopt it and focus claude-kit on the wiring.

3The four pieces

① <Agentation />A dev-only React component you render once in your app. Draws the bottom-right toolbar; captures what you click.
② agentation-mcpA small local server. Receives annotations from the toolbar (HTTP :4747) and exposes them to Claude over MCP.
③ /kit-annotateThe claude-kit skill that installs ①, registers ② for Claude, and runs the fix loop. Your entry point.
④ the rule.claude/rules/react-annotate.md — teaches Claude the annotate → locate → fix → resolve loop and the honest limits.
ℹ

You already know most of this. ① is just a React component. ② is a localhost server like your Vite dev server. MCP is "a REST API whose client is Claude" — the same mechanism that already lets Claude read tool results. ③ a slash command. ④ a markdown instruction file.

4How it works, end to end

  Your dev app (localhost) ──<Agentation/> toolbar──▶ you click + note
            │  HTTP POST :4747
            ▼
      agentation-mcp  (local server + SQLite)
            │  MCP (stdio)
            ▼
      Claude Code  ──reads annotations, greps to the file, fixes, marks resolved
1
Set up onceRun /kit-annotate. It detects your framework, installs agentation, registers its MCP for Claude, adds the rule, and helps you drop <Agentation /> into your app entry.
2
Run your dev server & annotateStart the app, click the bottom-right toolbar, click a component, type your note (with intent + severity).
3
Claude reads the queueIt calls the Agentation MCP — either on demand (agentation_get_all_pending) or hands-free (agentation_watch_annotations, which waits for new notes).
4
Locate → fix → resolveFor each note: it acknowledges, finds the code (source pointer or grep by component/selector), makes the change, and marks it resolved with a one-line summary.
ℹ

Who drives what: you click and annotate (a real click in your browser); Claude reads and fixes. In hands-free "watch mode" you just keep annotating while Claude resolves them in the background.

5Real use cases + the exact commands

A. First-time setup on a React app
# preview what it will do — no changes
/kit-annotate --dry-run
# → React: yes (v19.2)  Framework: next  Entry: app/layout.tsx
#   would: npm i -D agentation · register MCP · write rule · wire <Agentation/>

# do it (asks consent, shows the layout diff before saving)
/kit-annotate setup

Then: restart Claude Code (loads the MCP), run your dev server, and npx agentation-mcp doctor to verify.

B. Fix a spacing bug you can see

App is running. You click the cramped admin card in the toolbar and type "spacing too tight — apply the 24px section rhythm." Then in Claude:

/kit-annotate review
# Claude: pulls the note → it's <AdminCard> (app/admin/AdminCard.tsx)
#   → opens the file, adjusts the spacing tokens → marks it resolved.

Or just say it in plain language: "review my UI annotations and fix them."

C. Hands-free "watch mode" — annotate while Claude fixes
/kit-annotate watch
# Claude calls agentation_watch_annotations in a loop.
# You keep clicking + noting across the page; each note is picked up,
# fixed, and resolved with a summary — until you say stop.

Great for a focused design-polish pass across a whole screen.

D. Check status / triage later
/kit-annotate status
# shows the .annotate config, runs `agentation-mcp doctor`,
# and lists active annotation sessions.
ℹ

/kit-annotate with no argument is smart: if the project isn't set up yet it runs setup; if it is, it starts a session. It also auto-activates on natural language like "set up visual annotation" or "watch mode for UI feedback."

6Honest limits (so nothing surprises you)

⚠

Dev-only. The toolbar is gated behind a dev check and stripped from production. You always annotate against the dev server.

⚠

Identity, not live values. Agentation gives Claude the component name + tree + a source pointer + your selector — not the live prop/state values. Claude greps to the file using those. (Capturing sanitized props/state was our original "build-your-own" idea; it's parked unless you find you truly need it day-to-day.)

⚠

React Server Components render on the server and have no client component to point at — a clicked server-rendered node falls back to a CSS selector. Client components annotate fully.

⚠

It's a third-party package. Agentation is open and source-available (PolyForm Shield license), installed as your dev dependency. The kit never copies its code — it only wires it up.

7Why we adopt Agentation instead of building our own

The earlier lesson — don't depend on a closed tool you can't inspect (that's why we dropped onUI) — still holds, but Agentation passes the test where onUI failed: it's source-available (you can read, pin, or fork it) and speaks standard MCP (inspectable). So the risk shrinks to ordinary "someone else maintains it," which we contain by keeping claude-kit's own layer — the command, the rule, your config — as the stable contract.

✦

Net: you get a working loop in days instead of weeks, hand the hard browser/React-internals work to a maintained package, and keep the door open to a kit-owned backend later only if you discover you need richer capture (like live prop/state values). Build-vs-adopt details are in the implementation plan.
